# Безопасность Korean with Madie — текущее состояние и план

Этот файл описывает дыры в безопасности, найденные при аудите, и шаги, чтобы их
закрыть. Часть из них **нельзя исправить только кодом** — нужны действия в
Firebase Console (создание аккаунтов, выпуск правил). Поэтому здесь — инструкция.

---

## 1. Пароли админа лежат открыто в `app.js`

В `app.js` есть массив `ADMINS` с логинами и **паролями открытым текстом**.
`app.js` — публичный файл; любой может открыть DevTools и прочитать пароль, после
чего получит полный доступ к админ-панели (создание/удаление контента, удаление
учеников, просмотр всех профилей). Этот же список ещё и пишется в
`shared/admins` в базе.

### Почему это нельзя «починить» одним коммитом
Сейчас админ входит **не через Firebase Auth**, а сравнением строки с массивом
`ADMINS`. Значит, когда админ пишет в `shared/*` или `chats/*`, у клиента **нет
auth-токена**. Любые строгие правила, требующие `auth != null`, заблокируют
работу админки. Поэтому требуется миграция (см. ниже), а не просто удаление
пароля.

### Что сделать (Console + небольшая правка кода)
1. В Firebase Console → Authentication создать реальные аккаунты для админов
   (например `madina@…`). 
2. Записать их UID в базу как `/admins/{uid} = true` (вручную в Console).
3. В коде `loginUser()` для админа вызывать `signInWithEmailAndPassword`, а
   «админство» определять не паролем, а проверкой `/admins/{uid}`
   (или Custom Claim). Удалить массив `ADMINS` и узел `shared/admins`.
4. Включить строгие правила (`database.rules.json`).

---

## 2. Правил безопасности нет в репозитории

`firebase.json` не содержит блока `database`, т.е. правила задаются вручную в
Console и, судя по тому, что неаутентифицированные записи сейчас работают,
**вероятно открыты на чтение/запись**. Это значит: любой может прочитать данные
всех учеников, накрутить/обнулить чужой XP, писать в общий контент (→ stored XSS,
см. п.3), удалять чужие комментарии.

### Что сделано
Создан **`database.rules.json`** — готовый безопасный шаблон правил. Он
**намеренно не подключён** к `firebase.json`, чтобы случайный `firebase deploy`
не сломал живое приложение (правила требуют авторизации, которой пока нет).

### Что сделать, чтобы включить
1. Сначала выполнить миграцию из п.1 (админы → Firebase Auth).
2. Для гостей включить **Anonymous Auth** (Console → Authentication →
   Sign-in method → Anonymous), и в коде вызывать `_auth.signInAnonymously()` для
   гостей, а `socialUserId()` сделать равным `auth.uid` (тогда гостевые лайки и
   комментарии проходят проверку `auth.uid === $uid`).
3. Добавить в `firebase.json`:
   ```json
   {
     "database": { "rules": "database.rules.json" },
     "functions": [ … как сейчас … ]
   }
   ```
4. `firebase deploy --only database` и проверить все сценарии (вход, чат,
   лайки/комменты как у залогиненного, так и у гостя, админ-публикация).

---

## 3. Stored XSS в контенте админа — ИСПРАВЛЕНО в коде

Текст постов ленты, заголовки/описания видео и кастомных уроков вставлялись в
`innerHTML` без экранирования. Это уже **исправлено**: весь такой текст теперь
проходит через `escHtml`, а строки в inline-обработчиках — через новый помощник
`jsStr`. Но пока правила БД открыты (п.2), злоумышленник может писать в
`shared/*` напрямую, минуя интерфейс, поэтому полноценная защита наступит только
после включения правил.

---

## Приоритет

1. Миграция админов на Firebase Auth (п.1).
2. Включить Anonymous Auth для гостей + перевести `socialUserId()` на `auth.uid`.
3. Выпустить `database.rules.json`.

До этого приложение работает как раньше — изменения кода в этом аудите
безопасности обратно совместимы и ничего не ломают.
